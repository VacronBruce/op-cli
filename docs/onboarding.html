Part 1

Set up your tools

Install once and everything downstream — designs, docs, tickets — plugs into the same agent.

01

Install Claude Code

The terminal-native coding agent. This is the foundation — Figma and op-cli both plug into it.

Need · a terminal Need · a Claude account (Pro/Max) or Anthropic API key Windows · Git installed ~2 min
Windows: install Git first. Claude Code's plugin installs (the Figma and op plugins clone Git repos) and its diff/commit features rely on Git. If you don't have it, install it, then reopen PowerShell:
PowerShell · Windows
winget install --id Git.Git -e
Or download it from git-scm.com. macOS ships Git with the Xcode Command Line Tools — you'll be prompted the first time it's needed.

Install (native — no Node.js required)

Terminal · macOS
curl -fsSL https://claude.ai/install.sh | bash
PowerShell · Windows
irm https://claude.ai/install.ps1 | iex

The native installer downloads the right binary, adds it to your PATH, and keeps itself updated in the background — no Node.js needed.

On a corporate Mac? If curl is blocked, the npm fallback still works: npm install -g @anthropic-ai/claude-code (needs Node 18+).
Tip: Run this in a normal PowerShell window — no admin rights needed. Open a new terminal afterward so claude is on your PATH.

First launch & sign in

Move into any project folder and start the agent. On first run it opens your browser to authenticate.

Start it
# open a NEW terminal first, then:
cd path/to/your/project
claude

Follow the browser prompt to log in with your Claude subscription (or paste an API key). That's it — you're in an interactive session.

02

Connect the Figma MCP

Lets Claude Code read your Figma designs — components, variables, layout — and turn frames into code.

Need · Claude Code (step 01) Need · a Figma account, signed in ~3 min

Recommended · Official Figma plugin (remote server)

The remote server is hosted by Figma and has the fullest feature set. Install the plugin from inside — or before launching — Claude Code:

Terminal
claude plugin install figma@claude-plugins-official

Then authorize the connection:

  • Restart Claude Code so it picks up the plugin.
  • Run /plugin, open the Installed tab, select figma.
  • Complete authorization — click Allow access in the browser that opens. Make sure you're signed into the right Figma account.
  • Run /plugin once more and confirm the status shows connected.

What comes with it · Figma skills

Installing the plugin bundles Figma's MCP skills automatically — pre-built instructions that teach Claude how to handle common Figma tasks, so you can just say “implement this design” instead of spelling out every step. You get, among others:

  • figma-generate-design — assemble full-page screens from your design system.
  • figma-use — create/edit frames, components, and variables on the canvas.
  • figma-code-connect — link Figma components to their code counterparts.
  • figma-generate-diagram / figma-create-new-file — spin up FigJam diagrams or blank files.
No plugin? You can download the skills manually from Figma's repo and ask Claude to unzip and install them — but the plugin route above is the easiest and keeps them updated.

Alternative · Add the remote server by hand

If you'd rather skip the plugin, register the hosted endpoint directly, then authenticate with /mcp:

Terminal
claude mcp add --transport http figma https://mcp.figma.com/mcp
# then, inside Claude Code:
> /mcp

Alternative · Local Dev Mode server (Figma desktop app)

For the on-device endpoint: open a design file in the Figma desktop app, switch to Dev Mode, enable the MCP server in the right sidebar, then register the local address:

Terminal
claude mcp add --transport http figma-desktop http://127.0.0.1:3845/mcp
Sanity check: in a Claude Code session, run /mcp — figma should be listed and connected. Paste a figma.com frame URL and ask Claude to read it.
03

Connect Google Drive

Give Claude Code read/search access to your Drive files — specs, PRDs, sheets — so it can pull context straight from your docs.

Need · a paid Claude plan (Pro / Max / Team / Enterprise) ~2 min
Different from Figma: Google Drive is an Anthropic-hosted connector you switch on in your browser — not a CLI plugin. Don't try to authenticate from inside Claude Code; the hosted Google connector doesn't support local OAuth.

Turn on the connector

  • In your browser, open claude.ai → Settings → Connectors.
  • Find Google Drive and click Connect.
  • Complete the Google sign-in and Allow access — make sure it's the right Google account.

Pick it up in Claude Code

The connector loads automatically the next time Claude Code launches — no command to run. Restart any open session, then confirm:

Inside Claude Code
> /mcp

Google Drive should appear in the list. Try asking Claude to “search my Drive for the sprint PRD.”

No paid plan? The hosted connector needs Pro or above. Self-hosted community MCP servers exist as a workaround, but the official connector is the supported path.
04

Install op-cli

The op CLI for OpenProject sprints, boards, and tickets — plus its Claude Code plugin (/op:openproject, /op:standup, …).

Need · an OpenProject API key On VPN · openpr.epochbase.com must be reachable ~3 min

Get your API key first

The installer will ask for an OpenProject API token. Create one in the web UI:

  • Log in to openpr.epochbase.com, then click your avatar (top-right) → Account settings.
  • In the left sidebar, open Access tokens.
  • On the Provider tokens tab, under API, click + API token.
  • Give it a name you'll recognize (e.g. op-cli) and click Create.
  • Copy the token immediately — OpenProject shows it only once. Paste it when the installer prompts.
Treat it like a password. Anyone with the token can act as you in OpenProject. op-cli stores it in ~/.oprc (chmod 600). To rotate later: create a new token, then op setup --api-key=<new-key>.

Run the one-line installer

Terminal · macOS
bash <(curl -fsSL https://github.com/VacronBruce/op-cli/releases/latest/download/install.sh)
PowerShell · Windows
irm https://github.com/VacronBruce/op-cli/releases/latest/download/install.ps1 | iex

The repo is public, so this needs no GitHub login. The installer will:

  • Detect your platform and download the right op binary.
  • Prompt for your API key and write ~/.oprc (optionally a default project + sprint).
  • Add op to your PATH and enable shell completion.
  • Install the op Claude Code plugin if claude is present.
Windows note: no Git Bash or WSL required — the PowerShell line above handles everything. Git Bash / WSL users can run the macOS curl line instead.
Rotate the sprint each cycle: op setup --sprint="App_06/15/2026". Re-run the installer one-liner anytime to refresh both the binary and the plugin.
PM

For project managers · op-cli in practice

You rarely need to memorize flags — the /op: slash commands take plain English inside Claude Code. Here's what to reach for, by moment in the sprint.

Two ways to drive it. Type a /op: command in Claude Code and describe what you want, or run the raw op command in your terminal. Both hit the same OpenProject. Start with the slash commands.

Every morning · standup & status

/op:standup
Your daily digest — sprint progress, blockers, team work by person, and risks in one briefing.
op board
The current sprint board (kanban). Add --status=blocked or --component=android to filter.
op my team
All team items grouped by person — who's carrying what.
op blocked
Just the blocked items in the sprint.

Planning the next sprint

/op:sprint-prepare --project app --prd ./prd.md
Creates the next sprint, intakes tickets from IDs / a PRD / a Figma link, and auto-moves high-priority backlog + bugs. Pauses for your confirmation between phases.
op backlog --unestimated
Backlog items still missing estimates — your grooming list.
op backlog --priority p0,p1
The highest-priority backlog, so nothing urgent slips.
/op:assign-components
Finds sprint tickets with no component and infers the right one from labels/keywords. Add --dry-run to preview.

Filing & grooming tickets

/op:file-bug CC button crashes when publishing on Android
Guided bug filing — collects repro / expected / actual / acceptance criteria and the right component & label, then creates it well-formed.
op create task "Fix login page" --assignee="Ken Peng" --priority=P1
Create a ticket directly. Bugs: op create bug "Crash on save" --priority=SEV1.
/op:ticket-prep 12345
PM self-review before business review — checks completeness, clarity, business justification, and acceptance-criteria quality, with rewrite suggestions.
op check --sprint
Readiness check across every ticket in the sprint (warns on missing components, estimates, criteria).

Reporting & closing the sprint

op sprint progress -v
Full sprint report for stakeholders. Drop -v for a compact summary.
/op:sprint-close --release "[iOS][ETV] 1.2.0"
End-of-sprint: assign a release version to every "Ready for Release" ticket and move them to Done.
op overview
Cross-project dashboard of your open work — widen with --projects=8 --sprints=5.

Just ask in plain English

When you don't know the command, describe the outcome:

Inside Claude Code
> /op:openproject what's blocked in the current sprint?
> /op:openproject show my team's work for this sprint
> /op:openproject create a SEV1 bug "Crash on save" for NTD+, assign Bruce, android
> /op:openproject add tickets 101 102 103 to the current sprint
New sprint each cycle? Point op-cli at it once: op setup --sprint="App_06/15/2026". Everything above then scopes to that sprint automatically.
05

Auto-generate design specs with uSpec

Optional

Uber's open-source agentic system that writes design-system component specs into Figma in minutes. Skip this unless you maintain a reusable component library.

Who this is for. Design-systems teams documenting many components (anatomy, API, tokens, accessibility) at scale. For one-off screens, the Figma MCP + design skills in step 02 already cover you — you don't need uSpec.
Have already · Claude Code + Figma MCP (steps 01–02) Need · a top-tier model (Opus / GPT high tier)

You're two of the four pieces in already. uSpec adds its skills, a Figma Extract plugin, and a template library on top of the Claude Code + Figma MCP you set up above.

Install the skills

Terminal · in your project
npx uspec-skills init      # installs skills, writes uspecs.config.json
npx uspec-skills doctor    # verify the setup

Then, one-time setup

  • Add the uSpec Extract plugin from the Figma Community (captures component data to JSON).
  • Point uSpec at a Figma MCP + a Figma personal access token — the native Figma MCP from step 02 works.
  • Duplicate the uSpec template library from Community, move it to your team, and publish it.
  • Run the firstrun skill so uSpec learns your template component keys.

Everyday use

Generate the Markdown spec, then render sections onto the canvas:

  • create-component-md — builds the source-of-truth .md from the Extract JSON.
  • create-anatomy · create-api · create-property · create-color · create-structure · create-voice — render each spec section (the last maps a control to VoiceOver / TalkBack / ARIA at once).
Run it clean: one agent per Figma file, a fresh session per skill, and don't touch the canvas mid-run. Everything executes locally — no design data leaves your network. Full docs: docs.uspec.design.
✓

Verify everything works

Open a new terminal so all PATH changes are live, then run these.

Claude Code

  • claude --version prints a version, and claude doctor reports healthy.

Figma MCP

  • Inside a session, /mcp lists figma as connected.
  • Claude can read a Figma frame you paste a link to.

Google Drive

  • /mcp lists Google Drive after a session restart.
  • Claude can search or read a Drive file when you ask.

op-cli

  • op setup shows an all-[ok] checklist.
  • op board renders the current sprint; op my lists your items.
  • In Claude Code, /op:openproject what's blocked this sprint? responds.
Something red? Each failing [--] line from op setup prints its own fix. For Figma, re-run /plugin and re-authorize. For Claude Code, claude doctor pinpoints PATH or auth issues.

Part 2

Work effectively with Claude Code

The tools are installed. This is how to actually get good results — the loop to follow, how to prompt, which model, and where to keep a human in charge.

TL;DR. Plan first, code second (use Opus to plan). Keep a CLAUDE.md of team conventions. Prompt = stage + task + rules, with examples. Sonnet for coding, Opus for thinking. Sandbox Claude when it runs commands. Humans verify anything irreversible.
↻

The 4-phase loop

Every non-trivial task should pass through all four. Skip a phase, pay the price.

1
Context
Tell it about your code.
2
Plan
Use plan mode for non-trivial tasks.
3
Execute
Let it work, with guardrails.
4
Verify
Read the diff. Run the tests.
1

Plan first, code second

Start in plan mode (Shift + Tab to switch). Plan with a strong model like Opus.

Why it matters

  • Catches wrong assumptions before code is written.
  • Forces Claude to read the relevant files first — use @ to tag which ones.
  • Gives you a checkpoint to redirect or refine scope.
  • Makes the diff smaller and far more reviewable.

Examples

Prompt
Read @lib/auth.ts and propose a plan to migrate from JWT to session cookies.
Prompt
Add rate limiting to @/api/auth.ts.

Ask one question at a time. When you have enough, summarize back
what you'll build and wait for my confirmation before writing code.
Rule of thumb: for non-trivial tasks (more than one file), always plan first. For trivial edits — rename a variable, fix a typo — skip it.
2

Give it context with CLAUDE.md

A good CLAUDE.md is a one-time investment that fixes context for every future session.

No more repeating “don't add useless comments” or “match the existing style” in every prompt — write your team's conventions once and Claude reads them automatically. A solid starting point is the Andrej Karpathy skills.

CLAUDE.md · skeleton
# CLAUDE.md

## 1. Think Before Coding
State assumptions. Surface tradeoffs. Ask when unclear.

## 2. Simplicity First
Minimum code that solves the problem. Nothing speculative.

## 3. Surgical Changes
Touch only what you must. Match existing style.

## 4. Goal-Driven Execution
Define success criteria. Loop until verified.
3

The three-part prompt

Stage + task + rules — with examples when you can. (Source: Anthropic Claude 101.)

1 · Set the stage
Your role, objectives, and the context Claude needs. “You're on a FastAPI backend using SQLAlchemy 2.0 async + Pydantic v2.”
2 · Define the task
The action — write, analyze, build, refactor, debug. “Add a GET /users/{id}/preferences endpoint.”
3 · Specify the rules
Style, constraints, examples. “Match @routes/users.py. Use DI for the DB session. No new dependencies.”

All together

Prompt
You're helping me on a Python FastAPI backend that serves a mobile app.
We use SQLAlchemy 2.0 with async sessions and Pydantic v2.

Add a `GET /users/{id}/preferences` endpoint that returns the user's
notification settings (email, push, sms — all booleans).

Match the existing endpoint style in @routes/users.py. Use dependency
injection for the DB session. Return a Pydantic response model. No new
dependencies. Add a test in @tests/test_users.py following test_get_user.

Show, don't just tell (k-shot)

One or more worked examples beat a description. Claude matches naming, return types, and error handling from your samples far more reliably:

Prompt · 1-shot
Validate phone. Match this style:

validatePhone("+1-555-123-4567")
  → { ok: true, normalized: "+15551234567" }
4

Choosing a model

Sonnet for coding, Opus for thinking.

  • Sonnet — your default for day-to-day coding.
  • Switch to Opus when: you're stuck on a bug Sonnet can't crack, you need architectural reasoning across many files, or you're planning a large change before implementing.
  • Effort levels — medium for routine work; high when the task is genuinely complex or a first attempt already failed.
5

Guardrails: sandbox & humans

Convenience is not worth an rm -rf of your home directory.

Sandbox Claude when it runs commands

Letting Claude run shell commands on your laptop is convenient but risky. Isolate it from your host filesystem, credentials, and network so a bad prompt — or a prompt injection — can't delete files or exfiltrate ~/.aws/credentials.

Terminal
cd ~/work/payments-svc
docker sandbox run claude .

Keep a human in the loop

  • A human verifies before any merge to main.
  • A human verifies before any production deploy.
  • A human verifies any irreversible action — DB migration, data deletion, side-effecting API call.
  • Humans communicate with each other — avoid unattended agent-to-agent chains.
